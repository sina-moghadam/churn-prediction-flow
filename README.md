# Telco Customer Churn Prediction

An end-to-end MLOps pipeline that loads, cleans, and feature-engineers the IBM Telco Customer Churn dataset, trains and compares several classification models with MLflow, selects the best one, and serves it through a FastAPI prediction endpoint.

## Project Structure

```
final_prgg/
├── data/
│   ├── v1/
│   │   └── Telco_customer_churn.xlsx   # raw dataset (required, not generated)
│   ├── v2/                             # cleaned + encoded data (generated)
│   └── v3/                             # feature-engineered + scaled data (generated)
├── src/
│   ├── __init__.py
│   ├── config.py            # central configuration values
│   ├── logger.py            # shared logging setup
│   ├── mlflow_utils.py      # MLflow tracking URI / experiment setup
│   ├── data_loader.py       # loads the raw dataset (v1)
│   ├── preprocessing.py     # cleaning, imputation, categorical encoding -> v2
│   ├── features.py          # domain feature engineering, scaling -> v3
│   ├── train.py             # model tuning, training, MLflow logging
│   ├── evaluate.py          # metrics, threshold search, final test evaluation
│   ├── utils.py             # small IO helpers (CSV/JSON read/write)
│   └── api.py               # FastAPI prediction service
├── run_pipeline.py          # single entry point: runs the full pipeline end to end
├── requirements.txt
├── Dockerfile
└── .gitignore
```

## Pipeline Stages

Running `run_pipeline.py` executes five stages in order:

1. **Load raw data** — reads `data/v1/Telco_customer_churn.xlsx`.
2. **Preprocess** — drops identifier/leakage columns, imputes missing values, one-hot encodes categoricals, and writes dataset **v2**.
3. **Feature engineering** — adds domain features (active service counts, contract length, charge-per-tenure), scales continuous columns, and writes dataset **v3**.
4. **Train & compare models** — for each dataset version (v2, v3), tunes LogisticRegression, RandomForest, XGBoost, and CatBoost with `GridSearchCV` under stratified cross-validation, searches the best decision threshold, and logs every run to MLflow. The single best run across all models/versions (by a weighted combination of ROC-AUC, F1, accuracy, and recall) is recorded in `src/best_model_info.json`.
5. **Evaluate** — loads the best model and reports final metrics on the held-out test split, which is never touched until this step.

## Setup

### 1. Create and activate a virtual environment

```bash
python -m venv .venv
```

Windows (PowerShell):
```bash
.venv\Scripts\activate
```

macOS/Linux:
```bash
source .venv/bin/activate
```

### 2. Install dependencies

```bash
pip install -r requirements.txt
```

### 3. Add the raw dataset

Place the raw Excel file at:

```
data/v1/Telco_customer_churn.xlsx
```

## Running the Pipeline

From the project root:

```bash
python run_pipeline.py
```

This runs all five stages described above. MLflow tracking data is written locally to `mlruns/`.

## Serving Predictions

Once the pipeline has produced a best model, start the API:

```bash
uvicorn src.api:app --host 0.0.0.0 --port 8000
```

### Health check

```
GET /health
```

Returns the currently loaded model name and dataset version.

### Predict

```
POST /predict
```

Accepts a single JSON object, a list of objects, or `{"records": [...]}`, using the same raw column names as the original dataset (minus identifiers). Returns, for each record, the predicted class, decision threshold, and churn/stay probabilities.

## Running with Docker

Build the image:

```bash
docker build -t churn-app .
```

Run the full training pipeline (default command):

```bash
docker run churn-app
```

Serve the API instead:

```bash
docker run -p 8000:8000 churn-app uvicorn src.api:app --host 0.0.0.0 --port 8000
```

## Notes

- `data/v2` and `data/v3` are generated by the pipeline and can be regenerated at any time by rerunning `run_pipeline.py`.
- `mlruns/` and `.venv/` are excluded from version control.
- The decision threshold used at inference time is the one that maximized F1-score on the validation split during training, not a fixed 0.5 cutoff.
